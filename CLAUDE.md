# 项目重构宪法

## 绝对禁止（ZERO TOLERANCE）
1. **禁止引入新依赖** —— 除非现有依赖完全无法满足，且经过显式确认
2. **禁止改变现有 API 接口** —— 所有公开函数/类的签名必须保持向后兼容
3. **禁止添加未要求的功能** —— 只重构，不扩展。如果看到可以"顺便做"的事情，忽略它
4. **禁止删除现有测试** —— 只能新增或修复测试，不能删除
5. **禁止使用实验性/最新技术** —— 优先使用项目中已存在的技术栈和模式

## 重构原则（必须遵循）
1. **防御式编程** —— 所有输入必须验证，所有边界必须处理
2. **错误传播** —— 错误必须显式返回，不能静默吞掉（no bare `except:`）
3. **资源管理** —— 文件、网络、数据库连接必须显式关闭或使用上下文管理器
4. **日志完整** —— 关键操作必须有日志，错误必须有堆栈信息
5. **类型安全** —— 新增/修改的函数必须加类型提示

## 质量标准（重构完成 checklist）
- [ ] 所有现有测试通过
- [ ] 新增边界情况测试
- [ ] 错误路径测试覆盖
- [ ] 代码审查：无 magic number，无硬编码路径，无重复代码
- [ ] 性能：不引入明显性能退化（>10% 需说明）

## 技术栈（只能使用这些）
- Python 3.10+
- FastAPI / Flask（现有框架）
- SQLAlchemy / 现有 ORM
- pytest
- 现有依赖列表（见 requirements.txt）

## 决策规则
当面临选择时，按以下优先级：
1. 最简单的方式（KISS）
2. 最显式的方式（不要隐式魔法）
3. 最可测试的方式
4. 最符合现有代码风格的方式

## 禁止的"聪明"做法
- 元编程（除非现有代码已大量使用）
- 动态属性/方法
- 重载运算符
- 复杂的继承链
- 全局状态修改
