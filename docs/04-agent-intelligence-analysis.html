Agent 智能水平分析与提升方案

对 Agent Swarm 系统中各 Agent 的"大脑"层进行全面审计，识别瓶颈并给出改进路线图。

核心结论

基础设施（容器池、A2A 协议、注册中心、熔断器）搭建完整，但 Agent 的"大脑"是空心化的——prompt 简陋、无规划、无验证、无协作、串行执行。称之为 "swarm" 名不副实，本质是一个串行的"LLM + file tools"远程调用。

一、Orchestrator 层问题（编排器）

P0 没有任务规划（Planning）

orchestrator.py 的 system prompt（L26-59）只说"分析请求→选 agent→dispatch→finalize"，但没有结构化的推理框架。没有要求 LLM 先输出任务分解计划，直接就跳到 dispatch。

当前：用户说"做一个前后端分离的 TODO app"→ orchestrator 直接 dispatch frontend + backend，任务描述靠 LLM 临场发挥。

应该是：先输出结构化计划（需求分析、技术选型、API 契约定义、前后端任务拆分），然后基于计划 dispatch。

P0 纯串行执行，没有并行

_tool_dispatch_agent 使用 blocking=True（L351），每次 dispatch 一个 worker 就阻塞等待完成。前端和后端无法并行。

一个"swarm"系统，agent 却是一个一个串行跑的。

P0 没有 Agent 间协作

Workers 互相完全不知道对方在干什么：

  • 前端 agent 不知道后端 agent 的 API 定义
  • 后端 agent 不知道前端 agent 需要什么接口
  • 没有 shared context / API contract / design doc

缺失：一个 contract 定义环节（orchestrator 先生成 API spec，然后同时传给前端和后端）。

P1 没有质量验证

Orchestrator 的 finalize 工具（L434-437）只是把 LLM 输出的 summary 原样返回。没有验证产物文件是否存在、代码能否运行、前后端接口是否匹配。

上下文管理粗糙

  • max_iterations = 15（L218）硬编码，不管任务简单还是复杂
  • 最后 3 轮强制 finalize（L225-231），粗暴打断
  • max_tokens=4096（L241），对复杂任务严重不足
  • temperature=0.1（L239），过于保守

二、Worker 层问题（执行 Agent）

P0 System Prompt 极其简陋

三个 agent 的 prompt（worker.py L80-117）每个只有 ~10 行，内容是泛泛的"你是前端/后端/通用 agent，用 write_file 写文件"。

缺失：领域知识注入、few-shot examples、输出质量要求、代码规范约束、reasoning 指引。

P1 工具集太弱

Worker 只有 4 个工具，且 path 处理有严重 hack：

工具问题
write_filePath(args["path"]).name 强制只取文件名（L364），无法创建子目录
read_file同样被 strip 到文件名，只能读当前 role 目录
list_directory无参数选择，硬编码只列 role 目录
run_command没有安全限制，输出截断到 5000 字符

缺失：代码搜索、依赖安装、代码格式化、测试运行、git、HTTP 请求工具。

P1 没有自我验证

Worker 的 call_llm（L243-352）是一个朴素的 tool-calling 循环。LLM 写完文件就直接返回。没有"写完代码后运行测试验证"的步骤。

其他 Worker 问题

  • 没有迭代精化：20 轮迭代限制但无机制鼓励多轮优化
  • 没有进度反馈：全程阻塞，不汇报中间状态
  • 没有错误恢复：tool call 失败无结构化重试

三、架构级问题

P0 大量死代码 / 未接线模块

模块状态
src/resilience/circuit_breaker.py完整实现，从未被引用
src/adapters/ (OpenAI/CLI/MCP)完整实现，未接入 orchestration
src/registry/ (Redis)完整实现，未用于 agent 发现
src/swarm_sdk/SDK 客户端，未使用
src/gateway/注册 API，未接入
main.py L57AdapterManager(registry=registry) — 参数不存在，会报错

其他架构问题

  • 没有错误恢复/重试：Worker 失败后 orchestrator 不 retry
  • 没有记忆/学习：每个任务从零开始，不保留模板或经验
  • 阻塞式 A2A：只使用 blocking 模式，无流式

四、提升路线图（按优先级）

优先级改进方向影响难度
P0Orchestrator 加 Planning 阶段：先生成结构化任务计划（API contract + 技术选型 + 子任务拆分），再 dispatch极大提升协作质量中
P0并行 dispatch + 共享 context：前端/后端并行运行，共享 API contract 文档真正的 swarm，速度翻倍中
P0Worker prompt 大幅增强：注入领域知识、best practice、输出规范、few-shot examples代码质量直接提升低
P1Worker 加自我验证环节：写完代码后自动运行/检查减少不可用产物中
P1去掉 path strip hack，允许子目录能做真实项目低
P1加 review agent 检查产出质量质量保证层中
P1接线死代码：circuit breaker、adapter manager、registry系统韧性低
P2非阻塞 A2A + 流式进度用户体验中
P2动态模型参数（根据任务复杂度调整）适应性低
P2跨任务记忆：保存成功的 prompt 和代码模板渐进提升高
P3Agent 间直接通信：前端 agent 能向后端 agent 提问真正的协作 swarm高
