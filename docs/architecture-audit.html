🔴 P0 — 设计级问题（5 个）

P0-1：两套 session 存储互不同步 最高优先级

根因：同一 session_id 的数据存在两张表里，各记各的，互不通信。

SessionManager (sessions 表)SessionService (sessions_v2 表)
记录messages（对话历史）
shared_context（计划上下文）		state（结构化状态）
events（事件流）
谁读内置 orchestrator（execute 恢复记忆）dashboard / /api/sessions/{id}/events
谁写/api/chat save()orchestrator 写 plan/state（但从不读回来）

后果：orchestrator 把 plan 写进 sessions_v2.state，下一轮只从 sessions 表读 messages/shared_context——它写的东西自己永远不读。"会忘记计划"就是这个原因。

orchestrator.py:497-508  → await self._session_service.update_state(sid, {"plan": {...}})  # 写入
orchestrator.py:338      → self._shared_context = session.shared_context                   # 从 SessionManager 读（不同表！）

修复：合并为单一 session 存储统一 API；或至少让 orchestrator execute() 从 SessionService 读回 state.plan。

P0-2：shared_context 恢复条件有 bug 直接导致"忘记"

根因：恢复 shared_context 的条件依赖 messages 非空，但 messages 为空不代表 shared_context 为空。

orchestrator.py:335-346
if session and session.messages:     # ← messages 为空就进入 else
    self._shared_context = session.shared_context   # 恢复 ✅
else:
    self._shared_context = ""        # ← 擦掉！❌

场景：新 session 第一轮，shared_context 有值但 messages 为空 → shared_context 被擦。

修复：条件改为 if session:，messages 有无单独处理。

P0-3：SHARED_DIR 是进程全局变量，并发请求互相覆盖 导致文件写错目录

根因：worker 的 reload_config() 改的是 os.environ["SHARED_DIR"]（进程级全局）。uvicorn 是单进程，两个并发请求的 reload_config 会互相覆盖。

worker.py:55-64  → reload_config() 改 os.environ["SHARED_DIR"]
worker.py:529    → execute_file_tool 读 os.environ.get("SHARED_DIR")

# 请求 A 设 SHARED_DIR=/task-A，请求 B 设 SHARED_DIR=/task-B
# 如果 A 的 tool 执行在 B 的 reload 之后，A 的文件写到 /task-B 里

后果：并发任务时，A 的产物写到 B 的目录，B 看不到 A 的产出。

修复：SHARED_DIR 从全局变量改为 per-request context（FastAPI 的 request.state 或 contextvar）。

P0-4：INSERT OR REPLACE 全行覆盖，并发丢数据 丢消息/丢事件

根因：两套存储都用 INSERT OR REPLACE 整行写入。两个请求操作同一个 session_id 时，后写的覆盖先写的。

sqlite_store.py:126  → INSERT OR REPLACE INTO sessions (session_id, messages, ...) VALUES (...)
service.py:144        → INSERT OR REPLACE INTO sessions_v2 (session_id, state, events, ...) VALUES (...)

场景：请求 A 写入 messages=[m1]，请求 B 同时写入 messages=[m1,m2]。B 先写，A 后写 → messages 变回 [m1]，m2 丢失。

修复：改用列级 UPDATE 或加锁；read-modify-write 包在事务里；加 asyncio.Lock per session_id。

P0-5：finalize 只检查文件存在，不检查内容 垃圾代码=完成

根因：review_artifacts() 只数文件个数，不打开、不解析、不验证。

orchestrator.py:76-92
files = [f for f in d.rglob("*") if f.is_file()]
if not files: missing.append(agent_type)  # 有文件就算通过

后果：Agent 写了个空的 index.html 或 print("hello") 也算通过审查。

修复：加内容检查（至少检查文件非空 + 基本语法校验）；或让 LLM 读文件后判断。

🟡 P1 — 中等问题（6 个）

P1-1：容器归还时不清理上一个任务的文件 跨任务产物串扰

return_container() 只重置 config 为 {}，不删文件。shared_output 目录随任务增多无限膨胀，多任务文件堆在同一 session 目录里互相干扰。

pool.py:282-308

P1-2：_tasks / _bg_tasks 字典只增不删 内存泄漏

worker 的 _tasks 和 _bg_tasks 只在 handle_send_message 添加，从不删除。每个 task dict 含完整历史 + progress 列表，跑 50 个任务后可能占几十 MB。

worker.py:235, 354

P1-3：DockerBackend 只暴露 1 个逻辑 target 没有真正 failover

candidates() 永远返回 [DispatchTarget(kind=DOCKER, ...)]。dispatcher 的 failover 逻辑遍历 candidates，但永远只有 1 个——重试只是对同一个逻辑 target 再试一次，碰巧可能拿到不同的物理容器。

backends.py:43-45 + dispatcher.py:118

P1-4：LLM 超时 300s 但 worker 线程不杀 僵尸线程

dispatcher 的 asyncio.wait_for 300s 超时后，worker 容器被归还，但 worker 内部的 LLM 调用线程仍在跑——归还后的容器可能继续写文件到新任务的目录。

dispatcher.py:187 + worker.py:507

P1-5：ContainerState.DEAD 定义了但从不赋值 僵尸容器循环

容器启动/就绪失败时从不标记 DEAD，坏容器被归还后反复循环——每次 checkout 浪费 30s 就绪检查超时。

pool.py:18（定义）/ pool.py:267-282（就绪检查）

P1-6：result_cache 用 task 文本做 key 相同 prompt 返回旧结果

缓存 key 是 (agent_type, hash(task_text))，不是 task_id。如果两个不同 task 恰好用相同 prompt，第二个直接返回第一个的缓存结果。

result_cache.py:27
