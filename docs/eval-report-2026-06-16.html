四、发现并修复的 3 个阻断性 bug

Bug 1（P0 致命）：容器归还清理删除了 swarm.db 已修复

根因：P1-1 修复（容器归还清理文件）执行 docker exec rm -rf /workspace/artifacts/*， 但 /workspace/artifacts 是 shared_output/ 的 bind mount—— 这个命令删除了宿主机的 swarm.db，导致所有 SQLite 表消失， orchestrator 写 session event 时 no such table: sessions_v2 崩溃。

修复：移除 _clean_container_artifacts。跨任务隔离通过 per-task SHARED_DIR (contextvar) 实现，不需要清理容器文件系统。

pool.py

Bug 2（P0）：SQLite WAL 模式下 CREATE TABLE 不刷盘 已修复

根因：_conn() 每次调用设 PRAGMA journal_mode=WAL。 在 WAL 模式下，CREATE TABLE 的 commit 写入 WAL 文件但不刷入主 db 文件， 导致外部检查看到 0 字节 db + 0 表。

修复：移除 WAL 模式（改用默认 rollback journal）+ _init_tables 显式 commit() + close()。 main.py lifespan 加表验证 + force re-creation 兜底。

sqlite_store.py + service.py + main.py

Bug 3（P1）：worker config 在容器归还时被重置为 {} 已修复

根因：return_container 把 config.json 重置为 {}， 导致 worker 下次 reload_config() 读到空配置，用默认的 glm-coding-plan（而非 glm-4.7）。

修复：不再重置 config.json——checkout 时写入新 config 覆盖旧值。

pool.py

五、当前仍存在的问题（不影响产物质量，影响任务完成）

问题：orchestrator finalize review 路径偏差 → 任务不结束 待修

现象：worker 写文件到 backend/backend/main.py（双层 backend 目录）， 但 orchestrator 的 review_artifacts 在 backend/ 目录下找文件—— 找到 backend/backend/ 子目录算"有文件"通过，但后续 finalize 的二次检查可能不匹配， 导致 orchestrator 反复 dispatch 修正路径，任务永远 running。

影响：产物质量没问题（文件能跑），但 task 状态不翻 completed。

修复方向：worker 的 execute_file_tool 应检查目标路径是否已有 role 前缀， 避免 backend/backend/ 双层。或 review_artifacts 递归检查。

六、调用链路（完整 trace）

用户 POST /api/chat {"message":"写一个 FastAPI..."}
  → Swarm 创建 task + session
  → orchestrator.execute() 进入 LLM tool-calling 循环：
    [1] LLM 调用 (GLM 4.7) → plan_task(analysis, subtasks)
    [2] LLM 调用 → dispatch_agent(backend-engineer, task)
        → Dispatcher → DockerBackend → pool.checkout(worker-0)
        → A2AClient send_message(blocking=False) → worker:9001
        → worker 启动 LLM tool-calling 循环：
            [step 0] assistant: 分析需求
            [step 1] tool write_file("backend/main.py", content)
            [step 2] tool write_file("requirements.txt", ...)
            ...
            [step 7] task status → completed
        → DockerBackend poll_task 检测到 completed → DispatchResult(success=True)
    [3] orchestrator 收到 result → 写 session event
    [4] LLM 调用 → finalize(summary)
        → review_artifacts 检查文件
        → ❌ 路径偏差（backend/backend/ vs backend/）→ 反复 dispatch
  → task 永远 running（但产物已落盘且可运行）
