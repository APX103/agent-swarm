A2A (Agent-to-Agent) 协议调研报告

v1.0.0 开放标准 · Apache 2.0 · Linux Foundation 管理 · 150+ 组织生产使用

1. 协议概览

A2A 是由 Google 发起并捐赠给 Linux Foundation 的开放标准，用于 AI Agent 之间的通信和互操作。核心原则：Agent 之间基于声明的能力交换信息，不共享内部状态、记忆或工具。

┌─────────────────────────────────────────────────┐ │ Layer 1: Canonical Data Model │ │ Task, Message, AgentCard, Part, Artifact │ ├─────────────────────────────────────────────────┤ │ Layer 2: Abstract Operations │ │ Send Message, Get Task, Cancel Task, ... │ ├─────────────────────────────────────────────────┤ │ Layer 3: Protocol Bindings │ │ JSON-RPC 2.0 | gRPC | HTTP+JSON/REST │ └─────────────────────────────────────────────────┘

规范来源

规范定义文件：spec/a2a.proto 是唯一权威定义，所有 SDK 必须从 proto 文件重新生成。

3. Agent 发现 & AgentCard

发现机制

  1. Well-known 端点：https://<base_url>/.well-known/agent.json
  2. Agent 注册中心：公共或私有注册表索引 AgentCard
  3. 去中心化发现：P2P 网络、IPFS、区块链（未来）

AgentCard 完整结构

interface AgentCard {
  // 基本信息
  name: string;                    // "Recipe Agent"
  description: string;             // 人类可读描述
  url: string;                     // Agent 服务 URL
  version: string;                 // "1.0.0"
  protocolVersion?: string;        // "1.0.0"
  documentationUrl?: string;

  provider?: {
    organization: string;
    url?: string;
  };

  // 能力声明
  capabilities: {
    streaming?: boolean;                // 支持 SSE 流式
    pushNotifications?: boolean;         // 支持 Webhook 推送
    stateTransitionHistory?: boolean;    // 暴露状态变更历史
    extendedAgentCard?: boolean;         // 支持认证扩展卡片
  };

  // 认证 (OpenAPI 风格)
  security?: string[][];
  securitySchemes?: Record<string, SecurityScheme>;

  // 交互模式
  defaultInputModes: string[];        // ["text/plain"]
  defaultOutputModes: string[];       // ["text/plain"]

  // 技能列表
  skills: AgentSkill[];
}

interface AgentSkill {
  id: string;              // 唯一标识
  name: string;            // 技能名称
  description: string;     // 技能描述
  tags: string[];          // 分类标签
  examples?: string[];      // 示例场景
  inputModes?: string[];
  outputModes?: string[];
}

4. Task 模型

interface Task {
  id: string;                       // UUID
  contextId: string;                // 关联相关 tasks/messages
  status: TaskStatus;               // 当前状态
  kind: 'task';
  history?: Message[];               // 对话历史
  artifacts?: Artifact[];           // 产出
  metadata?: Record<string, any>;
}

interface Artifact {
  artifactId: string;
  name: string;
  parts: Part[];
  metadata?: Record<string, any>;
}

状态说明

canceled任务被取消
failed任务执行失败
rejected任务被拒绝

unknown未识别状态
submitted │ ▼ working ╱ │ ╲ ▼ ▼ ▼ completed failed canceled │ ▼ input-required ──→ working ──→ ... rejected / auth-required (异常分支)

5. 消息格式

interface Message {
  messageId: string;           // UUID
  role: 'user' | 'agent';
  parts: Part[];               // 内容部分
  kind?: 'message';
  taskId?: string;
  contextId?: string;
  referenceTaskIds?: string[];  // 跨任务引用
  metadata?: Record<string, any>;
}

// Part — 联合类型（最小内容单元）
type Part = TextPart | FilePart | DataPart;

interface TextPart {
  kind: 'text';
  text: string;
}

interface FilePart {
  kind: 'file';
  file: FileWithBytes | FileWithUri;
}

interface DataPart {
  kind: 'data';
  data: Record<string, any>;   // 结构化 JSON 数据
}

JSON-RPC 2.0 封装

// 请求
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "message/send",
  "params": {
    "message": { "role": "user", "parts": [...], "messageId": "..." },
    "metadata": {},
    "configuration": { "blocking": true }
  }
}

// 成功响应
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": { /* Task 或 Message */ }
}

// 错误响应
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": { "code": -32001, "message": "Task not found", "data": {} }
}

7. 通信模式

模式 1：同步阻塞

Client → message/send (blocking: true) → Server 处理 → 返回完成的 Task/Message

模式 2：同步 + 轮询（非阻塞）

Client → message/send (blocking: false) → 返回 "working" 状态 Task Client → tasks/get (重复) → 更新状态 Client → tasks/get → 返回 "completed" 状态

模式 3：流式 SSE

Client → message/stream → SSE 事件流: 1. TaskStatusUpdateEvent (state: "submitted") 2. TaskArtifactUpdateEvent (append: false) — 首个分块 3. TaskArtifactUpdateEvent (append: true) — 后续分块 4. TaskStatusUpdateEvent (state: "completed", final: true)

模式 4：推送通知（Webhooks）

Client → message/send + pushNotificationConfig → 返回 Task Server → POST StreamResponse 到 Webhook URL - Authorization header - X-A2A-Notification-Token header

模式 5：多轮交互（Human-in-the-Loop）

Client → message/send → Task (status: "input-required" + Agent 提问) Client → message/send (同 taskId, contextId) → 继续 ... 循环直到 completed

9. A2A vs MCP

方面MCP (Model Context Protocol)A2A (Agent-to-Agent)
用途Agent ↔ 工具/资源Agent ↔ Agent
连接对象数据库、API、函数其他自主 Agent
交互风格结构化、通常无状态有状态、对话式、协作
发现工具描述（类函数调用）AgentCard (well-known 端点)
协议JSON-RPC over stdio/SSE/HTTPJSON-RPC 2.0 over HTTP(S) + SSE
Task 管理无原生 Task 模型完整 Task 生命周期（9 状态）
创建者AnthropicGoogle → Linux Foundation
它们是互补的！
User → [A2A] → Agent A → [A2A] → Agent B → [MCP] → Database
Agent 使用 A2A 与其他 Agent 通信，使用 MCP 调用自己的工具。

Python SDK 示例

# Server 端
from a2a.server import A2AServer
from a2a.types import AgentCard, AgentSkill

agent_card = AgentCard(
    name="My Agent",
    description="A sample A2A agent",
    url="http://localhost:9999",
    version="1.0.0",
    skills=[AgentSkill(id="chat", name="Chat", description="Chat capability", tags=["chat"])],
    capabilities=AgentCapabilities(streaming=True),
    defaultInputModes=["text/plain"],
    defaultOutputModes=["text/plain"],
)

server = A2AServer(my_agent_executor, agent_card=agent_card)
server.start()

# Client 端
from a2a.client import A2AClient
client = A2AClient("http://localhost:9999")
result = await client.send_message(
    message=Message(role="user", parts=[TextPart(text="Hello!")])
)

Docker Agent Swarm 实现要点：
1. 每个容器运行 A2A Server，在 /.well-known/agent.json 提供 AgentCard
2. 发现：中央注册服务 / DNS / 硬编码端点
3. 通信：容器间 JSON-RPC 2.0 over HTTP（http://agent-container:9999/）
4. 任务委派：message/send → Agent B → 返回 Task → tasks/get 轮询或 message/stream 流式
5. 每个 Agent 保持不透明：不暴露内部工具/记忆，仅声明 skills
