Docker 沙箱 & 多 Agent 架构调研

1. Docker Agent 沙箱模式

两种架构模式（LangChain/Harrison Chase 分类）

模式说明优缺点
Pattern 1: Agent 运行在沙箱内 Agent runtime + tools + 执行环境都在容器内 ✅ 接近本地开发体验 ❌ 迭代慢（需重建镜像）❌ API key 暴露风险
Pattern 2: 沙箱作为工具 推荐 Agent 在服务器上运行，通过 API 调用沙箱执行代码 ✅ 迭代快 ✅ 凭证安全 ✅ 并行沙箱 ✅ 按需付费 ❌ 网络延迟
我们的选择：Pattern 1 变体
由于我们需要环境隔离（前端/后端不同环境）且 Agent 需要长期运行，采用 Pattern 1 变体：Agent 运行在容器内，但 API key 通过安全方式注入，容器间通过 A2A 协议通信。

2. Warm Pool 预启动

Jazzmine SandboxPool 模式

预启动可配置数量的容器（pool_size），消除 1-3 秒冷启动延迟。

┌─────────────────────────────────────────────────────┐ │ Container Pool │ │ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ │ │ │idle │ │busy │ │idle │ │idle │ │busy │ pool=5 │ │ └─────┘ └─────┘ └─────┘ └─────┘ └─────┘ │ ├─────────────────────────────────────────────────────┤ │ Checkout / Return Cycle │ │ │ │ checkout() → │ │ Fast Path: 返回空闲容器（立即可用） │ │ Overflow: 池空但 overflow 预算可用 → 新建容器 │ │ Blocked: 信号量满 → 异步等待释放 │ │ │ │ return() → │ │ 清理 /workspace (tmpfs RAM-disk) → 零数据泄露 │ │ 使用计数 +1，超过 50 次则回收重建 │ └─────────────────────────────────────────────────────┘

3. 动态配置注入

# docker-py 配置注入
import docker

client = docker.from_env()
container = client.containers.run(
    image="swarm-agent:latest",
    environment={
        "TASK_ID": task_id,
        "AGENT_ROLE": role,           # "frontend" / "backend" / "orchestrator"
        "LLM_MODEL": model,           # "glm-coding-plan"
        "LLM_BASE_URL": base_url,     # "https://open.bigmodel.cn/api/coding/paas/v4"
        "API_KEY": api_key,            # 通过 docker secrets 更安全
    },
    volumes={
        "/shared/artifacts": {"bind": "/artifacts", "mode": "rw"},
        "/config/task.json": {"bind": "/etc/task/config.json", "mode": "ro"},
    },
    detach=True,
)

5. 安全加固

container = client.containers.run(
    image="swarm-agent:latest",
    nano_cpus=int(0.5 * 1e9),        # CPU 限制
    mem_limit="512m",                  # 内存限制
    memswap_limit="512m",             # 禁用 swap
    read_only=True,                    # 只读文件系统
    tmpfs={"/tmp": "size=100m"},       # 可写 tmp
    security_opt=["no-new-privileges", "seccomp=default"],
    cap_drop=["ALL"],                  # 丢弃所有 capability
    user="nobody",                     # 非特权用户
    pids_limit=100,                    # 进程数限制
    network="swarm-net",              # 自定义网络
    detach=True,
)

9. 多租户方案

目录隔离

/shared_output/
├── tenants/{tenant_id}/
│   ├── tasks/{task_id}/
│   │   ├── input/           # 任务输入
│   │   ├── workspace/       # Worker 工作空间
│   │   ├── artifacts/       # 产出（可下载）
│   │   └── state.json       # 任务执行状态
│   └── config/              # 租户配置
└── shared_libs/              # 公共库（只读）

隔离策略

维度策略
存储每租户独立 volume 子路径 /shared_output/tenants/{id}/
网络Docker 自定义网络，按需网络策略
凭证每租户独立 API key，容器启动时注入
资源docker-py nano_cpus + mem_limit 配额
上下文task_id + tenant_id 双维度隔离，session 基于 task_id
产物交付FastAPI 端点校验 tenant_id 后提供下载

产物分发流程

1. 用户提交任务 → FastAPI 生成 task_id 2. 创建目录 /shared_output/tenants/{tenant_id}/tasks/{task_id}/ 3. Orchestrator 编排 Worker Agent 4. Worker 写入各自子目录 5. Orchestrator 汇总产物 6. FastAPI 端点：GET /api/tasks/{task_id}/artifacts → 校验权限 → 返回文件列表 7. GET /api/tasks/{task_id}/download → 返回压缩包

11. 总结 — 我们的架构决策

关键决策汇总：
  1. Agent 框架：Google ADK（Apache 2.0，A2A 原生支持，自动上下文管理）
  2. 通信协议：A2A（JSON-RPC 2.0 over HTTP + SSE），a2a-sdk Python
  3. 沙箱：Docker runc（ Namespaces + cgroups，低开销）
  4. 容器池：Jazzmine SandboxPool 模式，预启动 5 个，异步 checkout/return
  5. 编排模式：Orchestrator-Worker（Orchestrator 通过 A2A 委派任务给 Worker Agent）
  6. 存储：共享 Volume 挂载，每租户/每任务独立子目录
  7. 入口：FastAPI（REST + WebSocket）
  8. 模型：GLM Coding Plan（OpenAI-compatible API）
