框架对比调研：LangGraph vs Google ADK

针对 Docker-based Agent Swarm + A2A 通信场景的深度对比分析

1. LangGraph

MIT (core) Proprietary (Platform) | Python, TS, Java | ~15k stars | v0.6+

1.1 核心概念

LangGraph 是一个低级别的图编排框架，灵感来自 Google Pregel 系统。工作流建模为有向图，使用消息传递。

State（状态）：图中流动的共享数据

  • Schema 选项：TypedDict（推荐）、dataclass、Pydantic BaseModel
  • 每个 state key 有独立的 reducer 函数：override（默认）、append（Annotated[list, add]）、自定义
  • 预构建 MessagesState + add_messages reducer 用于聊天历史

Node（节点）：纯 Python 函数，接收 state → 计算 → 返回 state 更新

  • 签名：(state, config, runtime) -> state_updates
  • 自动转换为 RunnableLambda
  • 必须设计为幂等（重执行时完全重放）

Edge（边）：基于当前 state 决定下一个节点的执行

  • 正常边：固定转换 graph.add_edge("A", "B")
  • 条件边：动态分支 graph.add_conditional_edges("A", route_func)
  • 执行模型：离散 "super-step"，并行节点在同一 step 运行
# LangGraph 基础示例
from langgraph.graph import StateGraph, START, END
from langgraph.graph import MessagesState

def orchestrator(state: MessagesState):
    # 分析任务，决定路由
    return {"next": "code_worker"}

def code_worker(state: MessagesState):
    # 执行代码任务
    return {"messages": [AIMessage(content="代码已写完")]}

workflow = StateGraph(MessagesState)
workflow.add_node("orchestrator", orchestrator)
workflow.add_node("code_worker", code_worker)
workflow.add_edge(START, "orchestrator")
workflow.add_conditional_edges("orchestrator",
    lambda s: s["next"],
    {"code_worker": "code_worker", END: END})
workflow.add_edge("code_worker", "orchestrator")  # 返回

graph = workflow.compile()

1.2 A2A 支持

⚠️ A2A 支持现状
LangGraph 的 A2A 支持在 LangSmith Agent Server 中提供，非 OSS 核心。
OSS 端需要使用 a2a-sdk[http-server] 作为 work-around。
LangChain 团队成员评价："OSS doesn't have a particular 'server' implementation yet"
  • LangSmith Agent Server 提供 A2A 端点：/a2a/{assistant_id}
  • 支持方法：message/send, message/stream, tasks/get
  • Auto-expose AgentCard：/.well-known/agent-card.json?assistant_id={id}
  • 需要：pip install "langgraph-api>=0.4.21" + LangSmith API key / LangGraph Cloud license

1.4 Docker 部署

LangGraph Platform（托管）：需要 license

自托管（OSS）：

  • LangServe（FastAPI 适配器）包装编译后的图
  • docker-compose：app container + PostgreSQL container
  • langgraph dev 本地开发
限制：LangServe 无内置 auth、无后台运行、无 cron。部署包装器是专有的。

    app_name="my_app",

    memory_service=my_memory_service,

    session_id="session456",

2.2 A2A 原生支持

✅ ADK 是 A2A 的参考实现
A2A 从 v0.1.0 起内置于 ADK，提供：
  • RemoteA2aAgent — 消费远程 A2A Agent
  • Server-side A2A 端点暴露（FastAPI-based）
  • 自动 AgentCard 生成
  • 完整任务生命周期管理

2.3 多 Agent 编排模式（ADK 2.0 Workflow）

from google.adk import Agent, Workflow

# 顺序执行
root = Workflow(edges=[("START", agent_a, agent_b, agent_c)])

# 并行执行
root = Workflow(edges=[
    ("START", agent_a, agent_b),   # 并行启动 A 和 B
    (agent_a, "merge"),            # A 完成后汇合
    (agent_b, "merge"),            # B 完成后汇合
])

# Orchestrator 模式（推荐我们的场景）
orchestrator = Agent(
    name="orchestrator",
    model="glm-coding-plan",
    tools=[AgentTool(frontend_agent), AgentTool(backend_agent)],
    instruction="分析用户请求，分派给合适的专业 Agent..."
)
frontend_agent = Agent(name="frontend", model="glm-coding-plan", tools=[...])
backend_agent = Agent(name="backend", model="glm-coding-plan", tools=[...])

orchestrator.sub_agents = [frontend_agent, backend_agent]

4. 决策结论

🏆 推荐：Google ADK + a2a-sdk

对于我们的场景（Docker 容器 Agent Swarm + A2A 通信 + orchestrator-worker 模式），推荐 Google ADK 作为 Agent 框架，配合 a2a-sdk 实现 Agent 间通信。

核心理由：

  1. A2A 是一等公民 — ADK 是 A2A 的参考实现，原生支持 RemoteA2aAgent、Server-side A2A 端点
  2. Docker 是一等公民 — FastAPI 入口，离线可用，无 license 限制
  3. 自动上下文管理 — 过滤、摘要、延迟加载，无需手动实现
  4. Per-Agent 配置 — 每个 Agent 独立模型/API key/工具
  5. Apache 2.0 — 完全开源，无厂商锁定

LangGraph 的优势场景：需要最大程度的图执行控制、已深度投资 LangChain 生态、复杂 DAG 分支。

我们的方案：ADK 负责 Agent 运行时 + 编排逻辑，a2a-sdk 负责 Agent 间通信协议，Docker + 自定义容器池管理负责沙箱隔离。
