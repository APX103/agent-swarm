外部 Agent 动态接入方案调研

Agent Swarm v0.2 方案设计 · 2026-06-13 · 纯调研文档，不涉及实现
问题定义 接口适配 服务发现 健康检测 健壮性设计 推荐架构 结论

1. 问题定义

团队中其他开发者构建了自己的 AI Agent，但接入方式各不相同：

CLI Agent

命令行启动，stdin 输入，stdout 输出。如 claude --print、codex、自定义 Python 脚本。

Interactive Agent

交互式 REPL，多轮对话，保持会话状态。如 ipython 式的 Agent。

Completion API Agent

提供 OpenAI 兼容的 HTTP 接口，如 POST /v1/chat/completions。

其他协议 Agent

MCP Server、gRPC 服务、WebSocket 实时流等。

核心需求：

  • 动态接入 — Agent 上线后，Orchestrator 自动感知并可用
  • 动态离线 — Agent 下线后，Orchestrator 自动移除，不再调度
  • 接口无关 — 无论原始接口是什么，都能被统一调用
  • 健壮性 — 单个 Agent 故障不影响整体服务

2. 接口适配：Agent Adapter 模式

核心思路：定义一个统一的 Agent 接口规范（Canonical Interface），然后为每种异构接口编写一个薄适配器（Adapter），将原生协议翻译为统一格式。

                    ┌─────────────────────┐
                    │   Orchestrator      │
                    │   (统一调用接口)     │
                    └─────────┬───────────┘
                              │
              ┌───────────────┼───────────────┐
              │               │               │
     ┌────────▼──────┐ ┌──────▼──────┐ ┌──────▼──────┐
     │ CLI Adapter   │ │ HTTP Adapter│ │ MCP Adapter │  ...

2.1 统一调用接口（Canonical Interface）

所有 Adapter 必须实现以下接口：

class AgentBackend(ABC):
    """所有 Agent 适配器的统一基类"""

    async def invoke(self, task: str, context: dict) -> AgentResult:
        """同步调用 — 发送任务，等待结果"""
        ...

    async def invoke_stream(self, task: str, context: dict) -> AsyncIterator[AgentEvent]:
        """流式调用 — 逐步返回事件"""
        ...

    async def health_check(self) -> bool:
        """健康检查 — Agent 是否存活"""
        ...

    @property
    def capabilities(self) -> AgentCapabilities:
        """能力声明 — 该 Agent 能做什么"""
        ...

@dataclass
class AgentResult:
    success: bool
    output: str                    # 文本输出
    artifacts: list[str]           # 产出的文件路径列表
    error: str | None              # 错误信息
    metadata: dict                # 扩展元数据

@dataclass
class AgentCapabilities:
    skills: list[str]              # 能力标签 ["coding", "frontend", "data-analysis"]
    input_modes: list[str]         # 接受的输入格式
    output_modes: list[str]        # 输出格式
    tools: list[ToolInfo]          # 可调用的工具列表

2.3 CLI Adapter 详细设计

CLI Agent 是最常见的异构接口。适配器的核心挑战是进程生命周期管理：

class CLIAdapter(AgentBackend):
    """将任意 CLI 程序适配为 AgentBackend"""

    def __init__(self, command: str, args: list[str], timeout: int = 300):
        self.command = command          # 如 "python3", "claude"
        self.args = args                # 如 ["--print", "--json"]
        self.timeout = timeout

    async def invoke(self, task: str, context: dict) -> AgentResult:
        proc = await asyncio.create_subprocess_exec(
            self.command, *self.args,
            stdin=asyncio.subprocess.PIPE,
            stdout=asyncio.subprocess.PIPE,
            stderr=asyncio.subprocess.PIPE,
        )
        try:
            stdout, stderr = await asyncio.wait_for(
                proc.communicate(input=task.encode()),
                timeout=self.timeout,
            )
            return AgentResult(
                success=proc.returncode == 0,
                output=stdout.decode(),
                artifacts=self._parse_artifacts(stdout.decode()),
                error=stderr.decode() if proc.returncode != 0 else None,
            )
        except asyncio.TimeoutError:
            proc.kill()
            return AgentResult(success=False, output="", error="CLI process timed out")

    async def health_check(self) -> bool:
        """检查 CLI 可执行文件是否存在"""
        try:
            proc = await asyncio.create_subprocess_exec(
                self.command, "--version",
                stdout=asyncio.subprocess.PIPE,
                stderr=asyncio.subprocess.PIPE,
            )
            await asyncio.wait_for(proc.communicate(), timeout=5)
            return proc.returncode == 0
        except (FileNotFoundError, asyncio.TimeoutError):
            return False
⚠️ 注意：CLI 适配不适合长时间运行的交互式 Agent。如果 CLI Agent 需要保持会话状态（如需要多轮对话），应考虑将其包装为一个常驻的 HTTP 服务，而非每次 spawn 新进程。

2.4 OpenAI API Adapter 详细设计

对于已有 HTTP 接口的 Agent，适配最简单：

class OpenAICompatibleAdapter(AgentBackend):
    """适配任何 OpenAI 兼容的 /v1/chat/completions 接口"""

    def __init__(self, base_url: str, api_key: str = "", model: str = "default",
                 timeout: float = 300):
        self.base_url = base_url.rstrip("/")
        self.client = httpx.AsyncClient(timeout=timeout)
        self.headers = {"Authorization": f"Bearer {api_key}"}
        self.model = model

    async def invoke(self, task: str, context: dict) -> AgentResult:
        system_prompt = context.get("system_prompt", "You are a helpful assistant.")
        resp = await self.client.post(
            f"{self.base_url}/v1/chat/completions",
            headers=self.headers,
            json={
                "model": self.model,
                "messages": [
                    {"role": "system", "content": system_prompt},
                    {"role": "user", "content": task},
                ],
                "temperature": 0.7,
            },
        )
        data = resp.json()
        content = data["choices"][0]["message"]["content"]
        return AgentResult(success=True, output=content, artifacts=[], metadata=data.get("usage", {}))

    async def health_check(self) -> bool:
        try:
            resp = await self.client.get(f"{self.base_url}/v1/models", headers=self.headers)
            return resp.status_code == 200
        except Exception:
            return False

3. 服务发现：Agent 如何被发现

A2A 协议定义了三种发现策略：

策略模式适用场景动态性
Well-Known URI
GET /.well-known/agent-card.json		 Pull（拉取） 已知地址的公开 Agent 静态
Service Registry
注册中心		 Push + Pull（推拉） 动态伸缩的 Agent 池 动态
Direct Config
配置文件 / 环境变量		 Static（静态） 开发环境、紧密耦合 静态

对于我们的需求（Agent 随时上线/下线），Service Registry 模式是唯一合适的选择。

3.1 注册中心方案对比

方案 一致性 复杂度 自动注销 实时通知 适用规模
SQLite 强（单节点） 极低 定时 sweep 轮询 开发 / 单机
Redis 最终一致（单节点强） 低 TTL 过期 Pub/Sub 推荐 数十 Agent
etcd 强（Raft 共识） 中 Lease 过期 Watch API 百+ Agent / K8s
Consul 最终一致 高 Health Check + Deregister Watch 异构基础设施
✅ 推荐方案：Redis。理由：DGX Spark 上 Redis 部署成本极低（docker run -d redis）；TTL 过期天然支持自动注销；Pub/Sub 提供实时通知；对数十个 Agent 完全够用。如果未来需要分布式，可迁移到 etcd。

3.2 Redis 注册中心工作流程

Agent 上线:
  Agent ──POST──▶ Gateway ──SET──▶ Redis
                                Key: agent:{id}         TTL: 30s
                                Key: agent:{id}:meta     (endpoint, capabilities)
                                └──PUBLISH──▶ "agent:online:{id}"

心跳续约 (每 10s):
  Agent ──POST──▶ Gateway ──EXPIRE──▶ Redis (TTL 30s → 续约)

Agent 下线 / 崩溃:
  心跳停止 30s → Redis TTL 过期 → Key 自动删除
                                └──PUBLISH──▶ "agent:offline:{id}"

Orchestrator 感知:
  Redis Pub/Sub ──▶ Orchestrator 收到事件 → 更新可用 Agent 列表

3.3 Agent 端接入方式（对朋友最友好的方案）

我们提供两种接入模式，从简到繁：

模式 A：零代码接入（适合已有 A2A/HTTP 接口的 Agent）

Agent 只需在启动后向 Gateway 发一个注册请求：

# Agent 启动后调用一次
curl -X POST http://GATEWAY:19000/api/v1/agents/register \
  -H "Content-Type: application/json" \
  -d '{
    "name": "张三的 Code Review Agent",
    "endpoint": "http://zhangsan-pc:8080",
    "protocol": "openai",              # "a2a" | "openai" | "cli" | "mcp"
    "skills": ["code-review", "testing"],
    "heartbeat_interval": 10           # 秒
  }'

# 每隔 heartbeat_interval 秒发送心跳
curl -X POST http://GATEWAY:19000/api/v1/agents/{agent_id}/heartbeat

# Agent 关闭前发送注销（可选，TTL 过期会自动处理）
curl -X POST http://GATEWAY:19000/api/v1/agents/{agent_id}/deregister

模式 B：SDK 接入（适合需要更深度集成的 Agent）

from swarm_sdk import AgentClient

client = AgentClient(gateway_url="http://GATEWAY:19000")

# 注册 + 自动心跳（后台线程）
await client.register(
    name="我的 Agent",
    endpoint="http://localhost:8080",
    protocol="openai",
    skills=["code-review", "testing"],
)

# 此后 SDK 会自动每 10s 发送心跳
# Agent 关闭时 SDK 自动发送 deregister

# 或者用 context manager（推荐）
async with AgentClient("http://GATEWAY:19000") as client:
    await client.register(name="...", endpoint="...", skills=[...])
    # ... Agent 正常工作 ...
    # context manager 退出时自动 deregister
💡 关键设计：即使 Agent 不发送 deregister（如进程被 kill、断电、网络中断），Redis TTL 过期也会自动清除注册信息，Orchestrator 通过 Pub/Sub 立即收到通知。这就是 lease-based liveness的核心优势——不需要依赖 Agent 的显式注销。

4. 健康检测：如何知道 Agent 是否还活着

三层防护确保 Orchestrator 不会调度到已死亡的 Agent：

4.1 第一层：TTL + 心跳（被动检测）

参数推荐值说明
心跳间隔10 秒Agent 每 10s 向 Registry 发送一次心跳
TTL30 秒（= 3 × 间隔）Redis key 的过期时间，允许 2 次心跳丢失的容错
连续丢失阈值3 次3 次心跳未收到才标记为可疑（避免网络抖动误判）

流程：

Agent 正常运行 → 每 10s 发心跳 → Redis TTL 续约到 30s → 状态: healthy
Agent 崩溃 / 断网 → 心跳停止 → 30s 后 TTL 过期 → Redis 删除 key + 发 Pub/Sub → Orchestrator 移除 Agent

4.2 第二层：主动健康检查（调度前验证）

即使 Registry 中标记为 healthy，在实际调度前也做一次轻量级探测：

async def pre_dispatch_check(agent: RegisteredAgent) -> bool:
    """调度前快速探测 — 确认 Agent 端点真的可达"""
    try:
        async with httpx.AsyncClient(timeout=5.0) as client:
            resp = await client.get(f"{agent.endpoint}/health")
            return resp.status_code == 200
    except (httpx.ConnectError, httpx.TimeoutException):
        return False
⚠️ 防止 "Zombie Agent" 问题：仅靠心跳不能保证 Agent 的业务能力正常。Agent 进程可能存活（心跳正常）但内部 LLM API 已失效、内存泄漏导致超时等。主动健康检查可以捕获这类"半死不活"状态。

4.3 第三层：Phi Accrual Failure Detector（自适应检测）

对于生产级部署，推荐使用 Phi Accrual Failure Detector（Hayashibara 等，2004），替代简单的固定超时：

  • 原理：基于心跳到达间隔的历史统计（均值 μ + 标准差 σ），计算当前"可疑度" φ
  • 公式：φ = −log₁₀(1 − F(t))，其中 F(t) 是正态分布的累积分布函数
  • 优势：网络抖动时自动放宽阈值（减少误判），网络稳定时自动收紧（快速发现故障）
  • 实现者：Akka Cluster、Hazelcast
  • 配置参考：阈值 φ = 8（约 10⁻⁸ 概率判错），基于最近 100 次心跳计算
💡 建议：MVP 阶段用简单的 TTL + 心跳就够了。生产部署时再升级到 Phi Accrual。

5. 健壮性设计：单点故障不影响全局

5.1 Circuit Breaker（熔断器）

为每个外部 Agent 配置独立的熔断器，防止故障扩散：

                    调度请求
                       │
                       ▼
              ┌─────────────────┐
              │   Circuit       │
              │   Breaker       │
              │                 │
              │  ┌───────────┐  │
              │  │  Closed   │──▶ 正常转发到 Agent
              │  │  (正常)    │  │
              │  └─────┬─────┘  │
              │        │        │
              │  失败率 > 50%    │
              │        ▼        │
              │  ┌───────────┐  │
              │  │   Open    │──▶ 立即拒绝（不实际调用）
              │  │  (熔断)    │  │
              │  └─────┬─────┘  │
              │        │        │
              │  等待 30s        │
              │        ▼        │
              │  ┌───────────┐  │
              │  │ Half-Open │──▶ 放行 2-3 个探测请求
              │  │  (探测)    │  │
              │  └───────────┘  │
              └─────────────────┘
参数推荐值说明
failureRateThreshold50%滑动窗口内 50% 失败即熔断
slidingWindowSize10-20 次调用统计窗口大小
minimumNumberOfCalls5至少 5 次调用才开始计算失败率（避免前 1-2 次就熔断）
waitDurationInOpenState30s熔断后等待 30s 再开始探测
permittedCallsInHalfOpen3半开状态最多放行 3 个探测请求
slowCallDurationThreshold15s超过 15s 的调用也算"慢失败"

5.2 优雅降级层级

当外部 Agent 不可用时，Orchestrator 按层级降级：

级别状态行为用户体验
Level 0 Healthy 正常调度外部 Agent 完整功能
Level 1 External Down 回退到内置 Docker Agent 功能完整，可能质量略低
Level 2 Pool Exhausted 返回缓存的历史结果（如果有类似任务） 数据可能过时
Level 3 All Down 返回明确错误 + 预估恢复时间 用户知道发生了什么

5.3 任务重试与调度策略

  • 调度前检查：先查 Registry 获取 healthy 的 Agent 列表 → 按技能匹配 → 按优先级排序
  • 失败重试：指数退避 + 抖动（1s → 2s → 4s → ...，最大 30s）
  • 备选调度：首选 Agent 失败后，自动尝试同技能的备选 Agent
  • 全局重试预算：每分钟最多重试 N 次，防止重试风暴
  • 任务超时：单个任务整体超时（如 5 分钟），超时后强制完成并返回已有产出

5.4 幂等性保障

防止重复调度（网络重试、Orchestrator 重启等场景）：

  • 每个任务有唯一 task_id
  • 调度请求携带 idempotency_key
  • Agent 端检查是否已处理过相同 key 的请求
  • 共享存储目录使用 task_id 隔离，天然防止文件冲突

6.2 Gateway API 设计

Gateway 是新增的轻量级服务，作为外部 Agent 接入的入口：

EndpointMethod用途调用者
/api/v1/agents/registerPOSTAgent 自注册外部 Agent
/api/v1/agents/{id}/heartbeatPOST发送心跳续约外部 Agent
/api/v1/agents/{id}/deregisterPOST主动注销外部 Agent
/api/v1/agentsGET列出所有在线 AgentOrchestrator
/api/v1/agents?skill=code-reviewGET按技能查询 AgentOrchestrator

💡 Gateway 可以独立于 Orchestrator 部署，也可以合并到现有 FastAPI 服务中（作为新的 router）。对于 DGX Spark 单机部署，合并更简单；未来分布式部署时再拆分。

  "instance_id": "unique-per-session"   # 区分同 Agent 的不同实例
}
TTL: 30s（每次心跳续约）

# 按技能索引（加速查询）
Key:   skills:index:{skill_name}
Type:  SET of agent_ids

# Pub/Sub 频道
Channel:  agent:online:{agent_id}        # Agent 上线通知
Channel:  agent:offline:{agent_id}       # Agent 下线通知
Channel:  agent:status_change:{agent_id} # 状态变更通知

6.4 2026 Agent 协议生态定位

我们的架构如何与业界标准协议协同：

协议角色我们在哪用
A2A (Google) Agent ↔ Agent 通信 Worker 内部通信（已实现）；推荐外部 Agent 的首选接入协议
MCP (Anthropic) Agent ↔ Tools 通过 Adapter 支持 MCP Server 类型的外部 Agent
AG-UI (CopilotKit) Agent ↔ 用户 UI 未来前端 WebSocket 实时交互层（可选）
OpenAI API LLM 调用标准 适配已有 OpenAI 兼容接口的外部 Agent

7. 结论与实施路线

7.1 核心结论

  1. Agent Adapter 模式是最灵活的接口统一方案。每个异构接口一个薄适配器，所有适配器实现相同的 AgentBackend 接口。5 种适配器（CLI、OpenAI API、MCP、gRPC、WebSocket）覆盖 99% 的现有 Agent。
  2. Redis + 心跳 + TTL是实现动态注册/注销的最简方案。Agent 上线注册、10s 心跳续约、30s TTL 过期自动注销。Pub/Sub 提供实时通知。
  3. Circuit Breaker防止故障扩散。每个 Agent 独立的熔断器，50% 失败率触发熔断，30s 后探测恢复。
  4. 优雅降级保证可用性。外部 Agent 不可用时回退到内置 Docker Worker。
  5. Gateway Service作为新增组件，提供注册/心跳/发现/间接调用 API。可合并到现有 FastAPI 中。
