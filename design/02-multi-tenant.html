1. 多租户架构

1.1 租户模型

采用 共享基础设施、逻辑隔离 的多租户模型：

维度方案说明
API 层tenant_id 参数每个请求携带 tenant_id，未指定则使用 "default"
任务隔离目录命名空间shared_output/tenants/{tenant_id}/tasks/{task_id}/
容器池共享池 + 按需复用容器不绑定租户，通过 config injection 动态分配
模型配置租户级别覆盖支持不同租户使用不同 LLM（API key、endpoint）
产物分发租户独立 ZIP + API 下载产物按任务归档，通过 API 带鉴权下载

1.2 请求流程

Client → POST /api/chat { message, tenant_id: "acme-corp" }
  → TaskManager.create_task(tenant_id="acme-corp")
    → shared_output/tenants/acme-corp/tasks/{task_id}/
      ├── frontend/   ← Worker 写入
      ├── backend/    ← Worker 写入
      └── _final/     ← 最终产物
  → Orchestrator.execute()
    → checkout container → inject config (含 tenant context)
    → Workers write to shared dir
  → TaskManager.complete_task()
  → Client → GET /api/tasks/{task_id}/download → ZIP

2. 上下文管理

2.1 Agent 上下文策略

Agent 是 任务执行型，不需要长期记忆。采用以下策略：

策略实现说明
短期上下文 LLM conversation buffer 仅保持当前任务的对话历史
自动压缩 Token 阈值触发摘要 当上下文超过阈值（如 16K tokens），Orchestrator 生成摘要替换旧消息
任务间隔离 每次 checkout 新容器 任务完成后容器重置，新任务在干净环境中执行
Orchestrator 上下文 精简式 Orchestrator 只看 Worker 返回的摘要和产物列表，不看实现细节

2.2 上下文压缩算法

# Orchestrator 的上下文压缩
class ContextCompressor:
    MAX_TOKENS = 16000
    SUMMARY_THRESHOLD = 12000  # 到达此值时触发压缩
    
    def compress(self, messages):
        if estimated_tokens(messages) < self.SUMMARY_THRESHOLD:
            return messages
        
        # 保留最近 N 条消息
        recent = messages[-5:]
        
        # 对旧消息生成摘要
        old_messages = messages[:-5]
        summary = llm.summarize(old_messages)
        
        return [{"role": "system", "content": f"历史摘要: {summary}"}] + recent

2.3 Worker 的上下文管理

Worker 在容器内运行，有自己的上下文窗口：

  • 输入：Orchestrator 发来的任务描述（通过 A2A 协议）
  • 工具调用历史：文件写入、命令执行的中间结果
  • 输出：最终结果摘要 + 产物文件列表
  • 任务完成后容器重置，所有上下文自动清除

3. 产物分发

3.1 产物生命周期

┌─────────────────────────────────────────────────────────┐
│  产物分发流程                                              │
│                                                          │
│  Worker → 写入 shared_output/tenants/{tid}/tasks/{id}/   │
│    ├── frontend/index.html                               │
│    ├── frontend/styles.css                                 │
│    └── backend/main.py                                     │
│                                                          │
│  TaskManager.complete_task()                               │
│    → 扫描目录，收集产物列表                                 │
│    → 发射 "complete" 事件（含产物列表）                     │
│                                                          │
│  Client → GET /api/tasks/{id}/artifacts                   │
│    → 返回产物文件列表 + 大小                               │
│                                                          │
│  Client → GET /api/tasks/{id}/download                   │
│    → 返回 ZIP 压缩包（所有产物）                           │
│                                                          │
│  自动清理（可选 cron）：                                    │
│    → 7天后删除已完成的任务产物                              │
└─────────────────────────────────────────────────────────┘

4. 安全考虑

⚠️ 当前版本安全边界
  • V1 不做用户认证/授权，依赖网络层隔离
  • 容器通过 Docker 内置隔离，限制 CPU/内存
  • 未来版本需添加：API Key 认证、租户配额、容器安全加固

4.1 容器安全加固路线

阶段措施优先级
V1（当前）资源限制（CPU/内存）✅ 已实现
V1.1--read-only 文件系统 + tmpfs高
V1.1--security-opt=no-new-privileges高
V2gVisor / Kata Containers中
V2NetworkPolicy 限制容器出站中
