2. 核心组件

2.1 FastAPI Server（API 层）

端点方法功能
/api/chatPOST接收用户消息，创建任务，启动编排流程
/ws/tasks/{task_id}WebSocket实时推送 Agent 执行事件
/api/tasks/{task_id}GET查询任务状态和结果
/api/tasks/{task_id}/artifactsGET列出任务产物
/api/tasks/{task_id}/downloadGET下载产物压缩包
/api/agentsGET列出可用 Agent 类型（AgentCard）
/api/healthGET健康检查（含容器池状态）

2.2 Task Manager（任务管理器）

  • 每个用户请求创建一个 Task，分配唯一 task_id
  • 维护任务状态机：created → running → completed/failed
  • 管理 WebSocket 连接池，广播事件到订阅者
  • 任务完成后收集产物，提供下载接口
  • 多租户隔离：通过 tenant_id 区分不同用户的任务和产物

2.3 Orchestrator Agent（编排 Agent）

  • 使用 Google ADK Agent 类 + OpenAI-compatible LLM（GLM Coding Plan）
  • System Prompt 定义编排策略：分析任务 → 选择 Agent → 分配 → 监控 → 汇总
  • 工具：通过 A2A Client 向 Worker Agent 发送消息
  • 工具：读取共享目录中的产物和日志
  • 工具：创建 AgentCard 描述的 Worker（触发容器池分配）
  • 上下文自动管理：ADK 内置（自动摘要、过滤不相关事件）

2.4 Worker Agent（工作 Agent）

  • 每个 Worker 运行在独立的 Docker 容器中
  • 使用 a2a-sdk 提供 A2A Server 端点（message/send, tasks/get）
  • 使用 a2a-sdk 的 AgentCard 声明自身技能
  • AgentCard 预配置在镜像中，包含支持的模型列表和技能描述
  • 启动时从环境变量读取：模型选择、API key、端口、任务信息
  • 产出写入 /workspace/artifacts/{agent_role}/（挂载的共享卷）
  • ADK 自动管理上下文：仅当前任务相关的记忆

2.5 Container Pool Manager（容器池管理器）

  • 启动时预创建 N 个容器（默认 5），保持 warm 状态
  • 异步 checkout/return 机制，带信号量限流
  • 分配时根据 AgentCard 选择合适的容器，注入配置（模型、API key）
  • 容器使用计数，超过阈值（50 次）自动回收重建
  • 共享 Volume 挂载：所有容器共享产物目录
  • Docker 网络隔离：自定义 bridge 网络 swarm-net

2.6 Shared Volume（共享存储）

/shared_output/
├── tenants/{tenant_id}/
│   └── tasks/{task_id}/
│       ├── frontend/     # 前端 Agent 工作目录
│       ├── backend/      # 后端 Agent 工作目录
│       └── _final/       # Orchestrator 汇总后的最终产物

3. 数据流

[用户] --POST /api/chat--> [FastAPI] │ ▼ [Task Manager] 创建 task_id 创建共享目录 │ ▼ [Orchestrator] (ADK Agent, GLM Coding Plan) │ ┌─────────┼─────────┐ ▼ ▼ ▼ [Container Pool] checkout(idle) → 注入配置 │ ▼ [Worker Container 1: Frontend Agent] Port 9001, A2A Server 写入 /workspace/artifacts/frontend/ │ ▼ [Worker Container 2: Backend Agent] Port 9002, A2A Server 写入 /workspace/artifacts/backend/ │ ▼ (Orchestrator 通过 A2A 轮询/流式获取状态) │ ▼ [Orchestrator 汇总] 读取产物，生成总结 写入 _final/ │ ▼ [Task Manager] 通知 WebSocket 客户端 提供下载链接

4. 预启动容器池设计

4.1 容器镜像

单一镜像 swarm-worker:latest，包含：

  • Python 3.13 + Google ADK + a2a-sdk + OpenAI SDK
  • 所有预定义的 AgentCard（frontend, backend, general, 等）
  • Skills 模板和 MCP 客户端
  • A2A Server 入口脚本

4.2 启动流程

[系统启动] │ ▼ [构建/拉取镜像 swarm-worker:latest] │ ▼ [创建 Docker 网络 swarm-net] │ ▼ [预启动 5 个容器] ← 全部 detach, waiting 状态 │ 容器内部运行一个等待脚本： │ "等待 /etc/swarm/config.json 写入后启动 A2A Server" ▼ [Container Pool Ready] │ ▼ [收到任务] → [选择 AgentCard] → [找到 idle 容器] │ ▼ [写入 /etc/swarm/config.json] (模型、API key、角色、端口) │ ▼ [容器内部检测到配置] → [启动 A2A Server on assigned port] │ ▼ [Container 注册到 Pool Manager] → [A2A 可通信] │ ▼ [任务完成] → [容器归池] → [清理工作目录] → [回到 waiting 状态]

5.2 通信流程

1. Orchestrator → GET http://worker-1:9001/.well-known/agent.json ← AgentCard (确认这是 Frontend Agent) 2. Orchestrator → POST http://worker-1:9001/ { "jsonrpc": "2.0", "method": "message/send", "params": { "message": { "role": "user", "parts": [{"kind": "text", "text": "请实现登录页面的前端代码..."}] } } } ← {"jsonrpc": "2.0", "result": {"id": "task-xxx", "status": {"state": "working"}}} 3. Orchestrator → GET http://worker-1:9001/ (tasks/get) ← {"status": {"state": "completed"}, "artifacts": [...]} 4. Orchestrator 读取共享目录产物，继续编排或汇总

6. 多租户设计

6.1 隔离维度

维度策略实现
存储目录隔离/shared_output/tenants/{id}/tasks/{id}/
凭证注入隔离容器启动时注入 tenant 的 API key
上下文任务隔离每个 task_id 独立 session
产物访问控制API 校验 tenant_id 后允许下载
并发容器池共享所有租户共享容器池，通过 task_id 隔离

6.2 产物交付流程

用户 A: tenant-aaa, task-001 → POST /api/chat → 创建任务 → Worker Agent 写入 /shared_output/tenants/tenant-aaa/tasks/task-001/ → GET /api/tasks/task-001/download → 校验 tenant-aaa 权限 → 返回 task-001 目录的 zip 包 用户 B: tenant-bbb, task-002 → 完全独立的目录和产物 → 只能访问自己的产物 管理视角: → GET /api/tenants → 列出所有租户 → GET /api/tenants/{id}/tasks → 列出租户所有任务

6.3 上下文管理策略

  • 任务级隔离：每个 task_id 有独立的 ADK Session
  • 自动压缩：ADK 自动摘要历史对话，保持上下文窗口可用
  • 任务完成清理：任务结束后，可配置保留时间，超时自动清理
  • Orchestrator 轻量化：Orchestrator 不携带 Worker 的执行细节，仅记录关键决策和结果摘要
